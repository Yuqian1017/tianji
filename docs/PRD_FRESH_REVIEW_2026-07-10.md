# 天机卷主 PRD Fresh Review（第二轮）

- 日期：2026-07-10
- 审查基线：`master@a3b0e0e`
- 审查对象：`docs/PRD.md` 及 Roadmap、Coverage Matrix、Source Manifest
- Reviewer：独立只读 fresh reviewer
- 结论：`HOLD`
- 严重度：5 个 P1、2 个 P2；无 P0/P3 finding

## 最强反对意见

PRD 已找回“教学 + 游戏 + 实战工具共享 Database”的方向，但阶段合同仍可能让团队在没有工具闭环、没有可消费 Database、无法识别重复猜题的情况下把 P0/P1 标为完成。同时，V0 被写成无界全库串行闸门，与“先做最小切片”和首版排除中医课程的决定不一致。

## Findings

### P1-1：无界 V0 会持续阻塞教学游戏主线

PRD 把 Product B 定为下一主线并排除首版中医课程，但 Roadmap 又要求每个 runtime 消费点全部入账，并让药典、妊娠、方剂和穴位研究继续串行阻塞 P1。生产风险隔离与首切片直接依赖数据应是硬门槛；隔离后的非首切片研究应进入并行 backlog。

当前处理：用户已明确要求本轮继续验证整个 Database，因此审计继续执行；产品阶段合同仍需在 P1 开工前改为“安全隔离 + 首切片依赖数据”，不能把本轮用户指令固化为永久串行门槛。

### P1-2：首个工具闭环同时属于 P1 和 P2

首个可玩版本写有 64 卦/六爻实战事件，但 P1 完成条件只要求课程、练习、进度和渡劫，首次工具接入又放在 P2。按统一产品意图，至少一个最小工具事件、复盘和掌握度回写应成为 P1 硬验收项。

### P1-3：P0 没有交付可消费 Database

P0 目前只要求文档、来源和抽象合同，没有要求版本化 schema、状态转换、查询合同、迁移规则和首批 `accepted` 数据。Database 不必先做 SQL 服务，但必须交付一个课程、题目、AI 与工具可以共同读取的最小 canonical 数据集及查询测试。

### P1-4：规范答案没有强制绑定流派与适用范围

PRD 允许冲突来源并存，却让评估层使用单一规范答案。知识与题目合同必须强制包含 `tradition/lineage`、`convention`、`applicability`、版本和接受状态，避免把版本或流派差异判成学习者错误。

### P1-5：掌握度、渡劫和反刷题没有算法合同

PRD 给出 5 至 8 题、80% 通过和“阻止重复猜题”，但没有定义题目暴露、重试、去重、延迟复测、遗忘、掌握度更新与实战表现权重。产品指标也缺基线、目标比例和样本窗口，因此教学效果目前不可验收。

### P2-1：Product A 完成度表述过高

当前不是完整 Product A。更准确的状态是“产品壳层、账户历史和部分确定性核心已实现”；姓名、择日、64 卦查询等旧范围未实现，多数解释层仍 blocked。PRD 应采用 prototype/module probe 表述，避免把局部结构通过写成产品完成。

### P2-2：Roadmap 与 Coverage Matrix 不是一致的当前快照

Roadmap 的“下一决策”混入多个历史时点，Coverage Matrix 又把“已有逐域 normalized core”和“尚无统一知识 schema”混写。当前状态与历史进度应拆开，保留单一下一步和单一权威状态字段。

## 反例

团队可以整理几份静态 JSON、制作四组课程和一套可重复作答的渡劫题，在不接排盘工具、没有 Database 状态机、也无法识别刷题的情况下满足当前 P0/P1 文字验收。阶段全部标绿，统一产品主张仍未被验证。

## 建议解除 HOLD 的最小路径

1. 把 V0 永久门槛改为当前生产风险隔离与首切片直接依赖数据；其余全库审计保留为并行 backlog。
2. 选定一个首切片，把最小工具实战、复盘和掌握度回写纳入 P1。
3. P0 交付最小 canonical schema、状态机、首批 accepted 条目和查询测试，并加入流派/口径/适用范围字段。
4. 写清掌握度、重试、去重、延迟复测和产品指标算法。
5. 把当前 Product A 改称可复用壳层与部分已验证核心，并清理 Roadmap 历史“下一步”。

本报告是产品合同审查，不改变当前用户明确要求的全库验证任务，也不对任何领域数据授证。
