## 2. 已修复问题

### 卦线位序反转

`XIANTIAN.binary` 原本已按初爻到上爻编码，例如震为 `100`、艮为 `001`，但旧 `buildResult()` 又执行 `reverse()`。这会把震/艮、巽/兑相关的互卦和变卦算错。

以报数 5、2 为例：本卦风泽中孚、初爻动。旧 runtime 错算互卦乾为天、变卦风天小畜；修复后互卦为山雷颐，变卦为风水涣。

### Compendium 例题自身错误

同一例题原写“兑初爻变巽，变卦巽为风”。逐爻复算为：兑 `110` 的初爻由阳变阴后成为坎 `010`，所以变卦应为风水涣。课程例题已经同步修正。

### 时间法使用公历近似

旧 runtime 直接使用公历月、日，和《梅花易数》“年月日为上卦，年月日加时为下卦”的农历语境及项目 Compendium 声明不一致。

现使用已验证接入的 `lunar-javascript@1.7.7` 转换农历月日、农历年支和时支。闰月因原典未另述，项目明确声明 `reuse_base_month_number`：闰六月仍以 6 计数。

### 手抄笔画表和 Unicode 码位估算

旧表覆盖有限且有确定错误，例如“亿”错记为 1 画；未收录字按 Unicode 码位位置估算 1-25 画，码位与笔画没有这种关系。

现从 Unicode 17.0 `Unihan_IRGSources.txt` 的 `kTotalStrokes` 机械生成 U+4E00-U+9FFF 全表：

- 20,992 字完整覆盖；`亿=3`、`億=15`、`龘=48`。
- 源文件 SHA256 固定为 `d1c817dd7db84295dab0643c277d97c2fa742c245f8824e6736c2a0935095325`。
- 生成器、Unicode License v3 和数据版本随库保存。
- 超出范围的字符显式拒绝，不再估算或把非汉字当 1 画。

Unihan 将 `kTotalStrokes` 标为 informative；它提供稳定、可追溯的数据口径，不代表唯一书写或术数笔画标准。

## 3. 原典字占与现代适配

《梅花易数》卷一的字占并非当前旧代码所称的统一逐字笔画法：一字按楷书左右画数，二至三字按字分组，四至十字主要按平上去入取数，十一字以上又改按字数。

当前产品保留的“前少后多、分组累加笔画”因此标为 `source_pinned_modern_adaptation`，不能作为原典字占规范答案。Compendium 和 UI 均已改写这一边界。

## 4. 解释层边界

旧代码把体用关系直接存为“吉/凶/利/泄”，并把互卦称为过程、变卦称为结果。结构计算正确不能证明现实事件会按这些标签发生。

现已：

- 设置 `chartStatus = validated_deterministic`。
- 数字和农历时间法设置 `castingMethodStatus = validated_declared_method`。
- 现代文字法设置 `castingMethodStatus = source_pinned_modern_adaptation`。
- 设置 `interpretationStatus = not_validated`。
- 体用只输出可复算的五行关系，不再输出吉凶 verdict。
- AI 不得把互卦、变卦或体用关系包装成事实预测，并禁止医疗、法律、投资等高风险行动建议。

Playwright 走真实登录与起卦路径复核：报数 5/2 在 `1440x900` 显示风泽中孚、山雷颐、风水涣；当前时间法成功生成农历口径盘；“天机卷”按 4+6+8 画生成雷水解。`390x844` 移动视口与桌面均无水平溢出，结构/解释状态和现代笔画法标签可见。
