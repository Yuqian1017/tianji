## 1. 当前算法来自哪里

直接上游是 `database/xuanxue/compendium-new/04-bazi/03-geju.md` 中自称“简化版”的伪代码，没有原典版本、页码、现代教材或校勘记录。它大致采用：

- 月令与日主同类或生助时加 40。
- 年、月、时三透干各加减 10。
- 任一地支见同五行根时加 15。
- 总差值以正负 20 分身强、中和、身弱。

Runtime 并没有忠实实现这段伪代码，而是另造一套规则：

| 项目 | Compendium 简化版 | Runtime |
|---|---|---|
| 月令 | 五行生克二分 | 十二长生分为 36/24/14/6 |
| 其他力量 | 三个透干各 10 | 透干记 1，所有藏干统一记 0.5，再折成 40 |
| 根 | 有根加 15 | 有根 16、无根 4 |
| 阈值 | 总差值 `>20 / <-20` | 总分 `>=58 / <38` |
| 输出 | 有正负的简化分 | 包装成 `0-100` 分 |

因此，这个 `/100` 既不是经典量表，也不是 compendium 示例算法的忠实实现。

## 3. 五行计数边界

当前 `wuxingCount` 只做：四个天干各记 1，每个藏干各记 0.5。它没有处理：

- 月令与节气深浅。
- 本气、中气、余气的不同权重。
- 透干、通根、会局、合化与刑冲后的变化。
- 寒暖燥湿和整体气势。

它可以作为“盘中出现了什么”的结构计数，不能作为五行旺衰、缺补建议或医学证据。

## 4. 产品处理

已实施最小风险修复：

1. 底层旧分数保留作历史兼容，新增 `heuristic_only` 和方法/限制元数据。
2. 八字 UI 与 AI 输入不再展示 `/100`，不再输出固定“用神方向”。
3. 五行图改名为“结构计数”，明确不代表旺衰。
4. `bazihealth` 返回硬 `blocked` gate；不生成脏腑风险、疾病描述、食疗、大运健康或 AI 健康请求。
5. 旧八字健康历史在加载时也转换为 blocked，不恢复旧结论。

`bazihealth` 不能通过导入中医 Skill 获得合法性。其前提“八字结构计数能预测脏腑与疾病”本身需要独立传统来源、现代医学安全审查和产品风险决策。
