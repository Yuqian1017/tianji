# 天机卷数据库与排盘 Validation Plan

- 状态：执行中
- 版本：0.1
- 开始日期：2026-07-09
- 当前优先级：继续全库逐域验证；统一改用典籍见证优先。相术先逐条校勘原文与旧扩写；中医以医经、本草、方书和医家为文化核心，药典/监管仅作现代安全与使用边界；视觉几何可靠性另轨验证
- 审计模式：Full audit，先做高风险分层抽查，再按结果扩展到全量
- 非目标：本阶段不新增教学/游戏功能，不在收集 findings 时顺手修复

## 1. 目标

验证天机卷中每一类数据能否支撑当前工具和未来教学系统。验证结果必须指出：

- 检查了什么具体单元。
- 使用了什么独立依据。
- 通过只证明到哪一层。
- 发现了哪些 mismatch、流派差异或证据缺口。
- 哪些内容可以继续使用，哪些需要修复、隔离或撤下。

“代码能运行”“SHA 一致”“compendium 里这样写”“AI 也这样回答”都不能单独证明来源正确。对于传统文化条目，本阶段验证“某版本是否确有此文、转述是否忠实、归属与异文是否清楚”，不把现代科学可验证性当作删除古籍内容的前提。

## 2. 验证对象

### 2.1 Runtime

- 排盘引擎：六爻、梅花、八字、紫微、奇门、风水。
- 传统养生计算：体质评分、子午流注、五运六气、八字健康。
- 视觉解释：望诊、面相、手相的结构化数据和 Prompt 事实。
- 共享基础：公历/农历、节气、干支、真太阳时、城市经度、AI 输入格式。

### 2.2 Database / Raw source

- `compendium-new` 的 70 个 Markdown 文件。
- TCM Skill 的 50 个 references 与 42 个原文文本。
- 当前 `src/modules/*/data.js`、`engine.js`、`prompt.js` 中的隐形数据库。
- 未来规范化条目、课程和题目使用的每一个事实。
- 传统条目的版本见证、原文定位、归属争议、异文、现代转述和运行时引用。

## 3. 正确性的四种合同

| 类型 | 例子 | 正确性标准 | 默认方法 |
|---|---|---|---|
| 精确确定性 | 公农历转换、节气时刻、干支、纳甲、飞星路径 | 给定口径后输出唯一、边界一致 | 全量枚举、独立实现复算、官方历表 |
| 流派确定性 | 子初换日、紫微安星派别、奇门起局、风水飞星 | 声明流派和版本后输出唯一 | 口径声明、同派参考、fixture 与属性测试 |
| 来源忠实性 | 口诀、断语、经典原文、古方、医家论述、案例 | 文本与可定位版本吻合，归属不夸大，解释与原文分开 | 逐条 source_ref、影印/转录核对、异文与冲突并列 |
| 医疗/安全 | 药物、方剂、剂量、禁忌、红线、穴位操作 | 现行来源、安全字段和消费边界完整 | 现行规范/药典/教材交叉，专业复核 |

解释性内容不强行判成宇宙唯一真理，也不因现代证据不足而从历史库删除。验证目标是来源真实、流派清楚、条件完整、内部一致；现代证据、安全和产品表达在独立层处理。

## 4. 证据等级

| 等级 | 名称 | 通过条件 |
|---|---|---|
| V0 | 未验证 | 只有内容或代码，没有验证证据 |
| V1 | 结构验证 | 文件、字段、ID、链接、类型与 schema 正确 |
| V2 | 内部一致 | 同库无矛盾，正反关系、范围、枚举和算法属性成立 |
| V3 | 独立复算 | 与独立实现、官方历表或同派可靠参考逐例一致 |
| V4 | 双重/专业审校 | 高风险项由两条独立证据或合格专业人员复核 |
| V5 | Runtime 回归 | 当前用户输入路径、保存结果和展示与已验证核心一致 |

SHA256 只属于导入完整性证据，不计入 V1-V5 的内容正确性。

## 5. 验证策略

### 5.1 全量优先

以下对象尽量全量验证：

- 有限枚举表：天干地支、五行关系、八卦、64 卦索引、十神、纳甲、六亲、六神、星曜映射、九宫飞星等。
- 可机械生成的日期范围：公农历、干支、节气边界、日柱连续性、时柱、真太阳时日期跨界。
- 安全硬规则：禁忌组合、毒性/妊娠标识、红线映射和产品中出现的全部具体药名/方名/剂量/穴位操作。
- 拟进入系统的传统知识条目：每条至少解析到作品、见证、定位和原文；无出处项不得进入 accepted/runtime。

### 5.2 分层抽查

解释性条目、案例和长篇知识先做分层抽查：

- 每个文件至少覆盖开头、中段、末段及所有高风险小节。
- 每个领域覆盖实体、关系、规则、例外和案例。
- 抽查一旦出现系统性错误、来源虚构或漏条件，该文件/子域升级为全量审计。
- 抽查通过只给“样本未发现问题”，不授予整个领域 V3/V4。

### 5.3 双轨复算

算法不能只用它依赖的同一个库验证自己。每个核心输出至少需要：

1. 项目引擎结果。
2. 独立算法或独立实现结果。
3. 对关键边界再用官方历表、权威工具或人工推导确认。

若两个实现共享同一数据表或依赖，只算一条证据。

## 6. 八字首轮计划

### 6.1 必须先冻结的口径

- 年柱以立春还是正月初一换年。
- 月柱以节还是中气换月。
- 日柱换日点：00:00 或 23:00 子初。
- 时柱使用标准时、地方平太阳时还是真太阳时。
- 早子/夜子是否分日。
- 大运顺逆、起运岁数和交运时间的算法版本。
- 输入时区、夏令时、经度和日期跨界规则。

没有这些声明，两个结果不同不一定能判断谁错。

### 6.2 首轮 30 例分层样本

| 组 | 数量 | 覆盖 |
|---|---:|---|
| 普通日期 | 6 | 不靠近节气/午夜，男女各半 |
| 立春边界 | 4 | 前后分钟、跨年柱 |
| 其他节令边界 | 6 | 月柱切换前后 |
| 午夜/子时 | 4 | 22:59、23:00、23:59、00:00 附近 |
| 闰日与月末 | 3 | 2 月 29 日、大小月跨日 |
| 真太阳时跨时辰/跨日 | 3 | 东西经度极端城市 |
| 大运方向与起运 | 4 | 阴阳年干 × 性别组合 |

首轮结果只决定风险级别：

- 任一四柱基础字段 mismatch：P1，立即扩大该规则的边界与范围测试。
- 同类 mismatch 出现 2 次及以上：视为系统性错误，暂停依赖该输出的 AI 解读和教学复用建议。
- 只有流派差异：补口径字段和 UI 声明，不能直接记成 bug。

### 6.3 后续全量/属性测试

- 连续日期的日柱必须按 60 甲子逐日递增。
- 同一天干下时干遵守五鼠遁表。
- 月柱在声明的 12 个节令边界切换，非边界不跳月。
- 年柱只在声明的换年点变化。
- 真太阳时校正前后必须保留可解释的时间差和日期变化。
- 大运序列方向、每运干支递进和起运年龄边界保持一致。

## 7. 其他域顺序

1. 已完成定性单元：六爻、梅花、共享《周易》、紫微、奇门、风水下卦，以及共享城市/民用时区当前产品域；解释层分别 blocked。
2. 相术来源忠实性：两份 raw 的 386 条非空行已逐行分类为 100 条结构行、286 条已裁决行、0 条无人裁决；7 个来源记录、37 段界定摘录、45 条传统条目和 41 条旧文裁决已建，534 项 0 fail。十二宫、五官、气色、三才纹、行星掌丘和西式辅助线已分层；未定位、冲突和现代扩写均显式 blocked。下一步不是重复清单，而是 accepted 主张的影印页逐条校勘和真实图片视觉测量验证。
3. 中医：50 个 reference 已全部进入 inventory，但这只证明盘点完整。下一轮以医经、本草、方书、医案和医家版本为文化核心，逐方逐药挂原文；106 项食药目录、药典和监管快照降为现代身份、安全和使用边界，不再承担传统内容主证据。
4. 相术视觉测量与其余 runtime：手指角度、掌丘伪测量和旋转依赖已修复；继续用真实图片样本独立验证 MediaPipe 姿态/尺度/透视/遮挡鲁棒性。视觉测量是否可靠与古籍是否有此说法分别出结论。
5. 其他术数与共享有限表：按 runtime 消费和教学风险继续逐项验证；所有解释层改用同一典籍见证合同。

## 8. 审计账本

每个验证单元记录在 `docs/DATABASE_VALIDATION_LEDGER.md`，字段包括：

- Case ID、领域、具体条目/输入、当前实现位置。
- 预期结果与 correctness contract。
- 独立依据、证据等级、命令或 artifact。
- 状态：`planned`、`in_progress`、`pass`、`fail`、`blocked`、`school_difference`。
- mismatch、根因、影响范围、修复批次和回归证据。

## 9. 修复冻结与升级规则

- 首轮审计先记录 findings，不边查边修。
- P0/P1 若阻断后续复算，可以在记录原因后打断冻结做最小修复。
- 两次修复失败后回到证据收集；三次失败后停止并审查架构或口径。
- 修复后必须用原输入同路径复测，并增加相邻边界用例。

## 10. 完成标准

首轮八字审计完成：

- 口径写清。
- 30 个分层样本有独立复算证据。
- 所有 mismatch 有根因或明确 evidence gap。
- 能判断当前八字引擎应保留、局部修复还是替换。

全数据库验证完成：

- 所有 runtime 数据消费点进入 inventory。
- 有限表和确定性核心达到 V3，关键用户路径达到 V5。
- 中医高风险消费项达到 V4 或被隔离。
- 每个拟进入系统的解释性条目都有具体典籍/版本/定位；原文、传统转述、现代边界和产品呈现不混层。
- Coverage Matrix 与 Source Manifest 反映真实验证状态。
